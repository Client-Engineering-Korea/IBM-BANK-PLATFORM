📋 시스템 개요

IBM Bank 코어 뱅킹 시스템이란?

은행의 핵심 원장(ledger) 관리를 담당하는 FastAPI 기반 백엔드 서비스로, 계좌 관리, 거래 처리, 고객 정보 관리 등 은행의 핵심 업무를 지원합니다.

주요 기능

🔐 인증 및 권한 관리

  • OAuth2 기반 토큰 인증
  • 역할 기반 접근 제어 (RBAC)
  • TELLER, BACKOFFICE 역할

💳 계좌 관리

  • 계좌 조회 (역할별 필드 제한)
  • 당좌대월 한도 설정
  • IBAN 기반 검색

👥 고객 관리

  • 고객 정보 조회 (PII 포함)
  • BACKOFFICE 전용 접근
  • 이름 기반 검색

💸 거래 처리

  • 계좌 간 내부 이체
  • 거래 내역 조회
  • 수동 거래 등록

💰 잔액 관리

  • 실시간 잔액 계산
  • 당좌대월 한도 검증
  • 거래 합계 기반

📊 데이터 플랫폼

  • ETL 파이프라인
  • 합성 데이터 생성
  • 테스트 환경 지원

데이터 모델

4개의 주요 테이블

  • users - 시스템 사용자 및 인증 정보 (username, hashed_password, role)
  • customers - 고객 정보 (customer_id, name, 기타 PII)
  • accounts - 계좌 정보 (account_id, iban, customer_id, overdraft_limit_eur)
  • transactions - 거래 원장 (transaction_id, account_id, booking_ts, amount_eur, type)

역할별 권한 요약

TELLER: 창구 직원용 - 고객 대면 업무 (잔액 조회, 이체)

BACKOFFICE: 운영센터용 - 관리 업무 (고객 정보, 한도 설정, 수동 조정)

🔒 보안 및 무결성

보안 특징

인증: OAuth2 토큰 기반 인증 (Bearer Token)
권한 관리: 역할 기반 접근 제어 (TELLER, BACKOFFICE)
비밀번호: SHA-256 해시로 안전하게 저장
CORS: 설정 가능한 origin 제한으로 XSS 방지
PII 보호: BACKOFFICE 역할만 고객 개인정보 접근 가능

거래 무결성

4가지 무결성 보장 메커니즘

  • 원자성: 이체는 데이터베이스 트랜잭션으로 처리 (BEGIN/COMMIT/ROLLBACK)
  • 이중 기록: 모든 이체는 출금(TRANSFER_OUT)과 입금(TRANSFER_IN) 두 개의 거래로 기록
  • 잔액 검증: 이체 전 잔액 + 당좌대월 한도 자동 확인
  • 타임스탬프: 모든 거래에 UTC 기준 타임스탬프 기록

프로덕션 환경 고려사항

현재 시스템은 SQLite 기반으로 개발/테스트 환경에 적합합니다. 프로덕션 환경에서는 다음과 같은 개선이 필요합니다:

  • 데이터베이스: PostgreSQL 또는 Oracle로 마이그레이션
  • 캐싱: Redis를 통한 잔액 캐싱으로 성능 향상
  • 메시징: Kafka를 통한 비동기 거래 처리
  • 모니터링: 거래 추적 및 감사 로그 시스템
  • 고가용성: 다중 인스턴스 배포 및 로드 밸런싱
